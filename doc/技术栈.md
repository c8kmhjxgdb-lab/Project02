# 星愿之子技术栈与工程路线

本文以当前仓库实际状态为准，整理《星愿之子》（Starchild2D）的技术栈、工程分层、运行时数据流和后续演进路线。旧版愿景中提到的 SoLoud、Faust、Dear ImGui、Lua 5.4 等内容尚未接入，后续如需采用应作为独立技术决策处理。

## 项目定位

- 游戏类型：C++17 2D 俯视角动作 RPG。
- 当前重点：世界探索、战斗、NPC 对话、情绪系统、天气昼夜、区域切换、秘密基地、建造、任务、玩具小游戏和存档。
- 资源策略：GLSL 着色器和 C++ 几何绘制为主，Lua 承载配置和对话内容，JSON 承载存档。
- 运行方式：CMake 将 `assets/shaders/` 与 `assets/scripts/` 复制到 `build/assets/`，可执行文件应从 `build/` 目录启动。

## 当前实际技术栈

| 领域 | 当前选型 | 用途 |
|------|----------|------|
| 语言 | C++17 | 引擎封装、玩法系统、主循环、存档和性能敏感逻辑 |
| 构建 | CMake + vcpkg manifest | 生成 Visual Studio 工程并安装依赖 |
| 窗口与输入 | SDL2 | 窗口、OpenGL 上下文、键盘鼠标事件 |
| 图形 API | OpenGL 3.3 Core + GLEW | 2D 绘制、SDF 角色、粒子、后处理 |
| 着色语言 | GLSL | 角色、敌人、弹幕、粒子、后处理等视觉效果 |
| 物理 | Box2D v3 | 俯视角零重力刚体、碰撞和查询 |
| 数学 | GLM | 向量、矩阵、摄像机与 GLSL 对齐的数据结构 |
| 脚本 | LuaJIT + sol2 | 对话、天气、家具、任务、玩具、能力等配置 |
| 序列化 | nlohmann/json | 存档读写与结构化数据转换 |
| 音频 | 未接入 | `music/` 资源已存在，源码层还缺音频系统 |

当前 `vcpkg.json` 依赖包括：`sdl2`、`glew`、`box2d`、`glm`、`sol2`、`luajit`、`nlohmann-json`。

## 推荐目标架构

仓库已经有 `Engine / Game / Utils` 的雏形，后续应演进为更明确的运行时架构：

```text
src/
├── Platform/              # SDL/OpenGL窗口、输入设备、文件系统、系统时间
├── Core/                  # Application、GameLoop、SceneManager、EventBus
├── Engine/                # 渲染、物理、音频、脚本、资源、摄像机
├── Game/
│   ├── Domain/            # 纯领域数据与实体
│   ├── Systems/           # 战斗、AI、任务、天气、情绪、建造等规则
│   ├── Controllers/       # 玩家、菜单、交互等输入控制
│   ├── Presentation/      # HUD、菜单、对话、小地图、世界渲染
│   ├── Services/          # 存档、区域、对话、音频等跨系统用例
│   └── Data/              # Lua配置、JSON存档、版本迁移
├── App/                   # 组合根
└── Utils/                 # 通用工具
```

详细分层和重构路线见 `doc/架构分层与重构路线.md`。

## 核心运行流

目标主循环应保持顺序稳定：

```text
poll events
scene.handleInput
fixedUpdate physics
gameplay systems update
collision resolution
cleanup
camera/minimap lateUpdate
render world
render UI
swap buffers
```

当前 `src/main.cpp` 已经隐含了这套顺序，重构时必须保留物理、碰撞处理、死亡清理、区域更新、小地图更新、后处理和 UI 绘制的相对顺序。

## 数据流示例：一次技能释放

1. SDL 捕获按键事件。
2. 控制层将按键转换为 `CastAbility` 意图。
3. 能力系统检查魔法、冷却、当前状态和目标方向。
4. 投射物或效果系统创建对应运行时对象。
5. 物理和碰撞系统推进位置并检测命中。
6. 战斗系统结算伤害、掉落、粒子和任务进度。
7. 视图层读取状态并绘制角色、弹幕、粒子、血条和 HUD。
8. 后续接入音频后，由事件触发技能音效和命中音效。

## 内容数据

```text
assets/
├── shaders/               # GLSL：角色、敌人、弹幕、粒子、后处理、装饰等
└── scripts/               # Lua：能力、天气、家具、任务、玩具、对话等
```

脚本和配置后续应通过 `Game/Data` 或 `Engine/Resource` 统一加载，玩法系统不应直接拼接资源路径。

## 音频路线

当前仓库已有 `music/` 资源，但代码层没有音频后端。推荐路线：

1. 在 `Engine/Audio` 中新增音频后端封装。
2. 技术选型优先考虑 SDL_mixer、MiniAudio 或 SoLoud，避免直接把音频 API 泄漏到玩法系统。
3. 先支持 BGM、SFX、淡入淡出、音量分组。
4. 再引入 `MusicDirector`，根据区域、天气、战斗状态和剧情事件切换音乐。
5. 玩法系统只发布事件，例如 `RegionEntered`、`SkillCast`、`EnemyKilled`、`WeatherChanged`。

## 构建与运行

```bash
cmake -B build
cmake --build build --config Release

cd build
./Release/Starchild2D.exe
```

Debug 构建：

```bash
cmake --build build --config Debug
```

Visual Studio 工程：

```bash
cmake -B build -G "Visual Studio 17 2022" -A x64
```

目前没有集成 `ctest` 单元测试。重构后建议优先为纯逻辑模块增加测试，例如存档序列化、任务推进、天气状态机、寻路和配置解析。

## 近期工程重点

1. 降低 `main.cpp` 密度，优先抽出菜单、HUD、世界渲染和输入处理。
2. 引入场景边界，用 `MainMenuScene / WorldScene` 取代 `AppMode` 分支。
3. 拆分 `GameState`，避免继续成为所有系统的共享大对象。
4. 将存档流程迁移为 `SaveService + Repository + Serializer + SnapshotBuilder + Applier`。
5. 补齐音频层，让 `music/` 资源真正进入运行时。
6. 将 CMake 从单 executable 源列表逐步演进为 `Engine`、`Game`、`App` 三个 target。

## 远期扩展

- Mod 支持：开放 Lua 配置、任务、对话和部分玩法参数。
- 多平台：SDL2 和 CMake 保持 Windows 优先，同时保留 Linux/macOS 移植空间。
- 自动化测试：先覆盖无渲染依赖的系统，再考虑渲染快照和脚本配置校验。
- 工具链：后续可加入资源打包、脚本校验、存档迁移检查和性能统计。
