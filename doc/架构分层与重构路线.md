# 星愿之子架构分层与重构路线

本文记录当前项目的推荐分层、核心原则、依赖方向和渐进式重构路线，作为后续重构与新增系统时的工程约束。

## 当前评价

项目已经具备清晰的原型基础：`src/Engine/` 承载渲染、物理、摄像机、脚本等基础设施，`src/Game/` 按玩法域拆分了世界、能力、AI、社交、情绪、建造、库存、任务、玩具和存档等模块。

主要维护风险集中在 `src/main.cpp`：入口、资源初始化、输入、主菜单、玩法规则、存档套用、物理更新、渲染排序和 UI 绘制集中在同一个文件中。这个结构适合阶段式快速验证，但继续扩张会导致回归风险高、初始化顺序脆弱、跨系统改动难以测试。

## 推荐分层

```text
src/
├── Platform/              # SDL/OpenGL窗口、输入设备、时间、文件系统
├── Core/                  # Application、GameLoop、SceneManager、EventBus、服务注册
├── Engine/
│   ├── Renderer/          # Draw2D、TextRenderer、RenderQueue、后处理、SDF渲染
│   ├── Physics/           # Box2D封装、碰撞查询、物理句柄
│   ├── Audio/             # 音频设备、BGM、SFX、混音与淡入淡出
│   ├── Camera/            # 2D摄像机
│   ├── Scripting/         # LuaVM、脚本绑定、脚本配置加载
│   └── Resource/          # Shader、Texture、Font、Script、Audio资源管理
├── Game/
│   ├── Domain/            # 纯数据与实体：Player、Enemy、NPC、Quest、Region、Item
│   ├── Systems/           # Combat、Ability、AI、Quest、Emotion、Weather、Time、Building
│   ├── Controllers/       # PlayerController、MenuController、InteractionController
│   ├── Presentation/      # HUD、DialogueView、MiniMapView、WorldRenderer、MenuView
│   ├── Services/          # SaveService、RegionService、DialogueService、AudioService
│   └── Data/              # SaveRepository、LuaConfigRepository、JsonSerializer
├── App/                   # 组合根：创建服务、装配场景、启动游戏
└── Utils/                 # 数学、随机、字符串、通用小工具
```

现阶段不要求一次性移动到完整目录结构。先在现有目录下按职责抽类，等边界稳定后再做目录迁移。

## 层职责

- 启动层：只负责初始化运行环境、创建 `Application`、进入 `run()`、退出清理。
- 循环层：固定事件、输入、更新、碰撞、清理、渲染、交换缓冲的顺序。
- 平台层：隔离 SDL、窗口、OpenGL 上下文、文件路径和系统时间。
- 引擎层：提供可复用能力，不感知具体游戏概念。
- 游戏实体层：保存游戏世界中的领域数据，避免直接持有渲染或平台资源。
- 游戏系统层：处理时间推进、战斗、AI、任务、天气、情绪等规则。
- 控制层：把输入和 UI 命令翻译为游戏意图。
- 服务层：组织跨系统用例，例如保存、读档、区域切换、对话启动、音频策略。
- 接口层：定义服务、资源、存储和平台抽象，减少具体实现泄漏。
- 数据访问层：负责 JSON、Lua、文件读写和版本迁移。
- 视图层：只读取状态并绘制画面，不承载玩法规则。
- 工具层：只放无状态或低耦合的通用工具。

## 核心原则

1. `main.cpp` 只做启动层  
   目标是把 `main.cpp` 缩到几十行：初始化 `Application`，调用 `run()`，处理返回码。

2. 主循环顺序显式化  
   当前物理、碰撞清理、渲染和存读档顺序很关键，必须沉淀为 `GameLoop` 或 `GameScene` 的固定流程：

   ```text
   poll events
   scene.handleInput
   fixedUpdate physics
   gameplay systems update
   collision resolution
   cleanup
   camera/minimap lateUpdate
   render world
   render UI
   swap buffers
   ```

3. 避免继续扩大 `GameState`  
   先将 `GameState` 拆为更小上下文，再逐步转入场景和服务：

   ```text
   WorldState      # 区域、地图、装饰、时间、天气
   PlayerState     # 血量、魔法、位置、童心、飞行、技能冷却
   CombatState     # 敌人、弹幕、掉落、粒子
   UIState         # 菜单、提示、对话、HUD
   RuntimeServices # renderer、physics、audio、scripting、resources
   ```

4. 玩法状态与视图状态分离  
   渲染器和 UI View 接收只读 ViewModel，不直接依赖完整 `GameState`。

5. 服务层承接跨系统流程  
   保存/读取、区域切换、对话启动、音频播放等跨多个系统的流程，不应散落在主循环。

6. 数据驱动要收口  
   Lua 配置、JSON 存档和资源路径由 Repository 或 Resource 层管理，玩法系统不直接拼路径。

7. 引擎不反向依赖游戏  
   `Engine` 可以暴露能力和接口，但不能知道 `Princess`、`Quest`、`WeatherSystem` 等具体玩法概念。

8. 渐进式重构优先  
   每次只抽一个明确边界，保持可编译、可运行，避免一次性搬动主循环导致行为回归。

## 依赖方向

理想依赖方向如下：

```text
App
 ↓
Core / Scene
 ↓
Game Controllers / Services / Presentation
 ↓
Game Systems
 ↓
Game Domain
 ↓
Engine Interfaces
 ↓
Platform / Third-party
```

约束：

- `Engine` 不依赖 `Game`。
- `Game/Domain` 不直接调用 SDL、OpenGL、Box2D 文件 API。
- `Presentation` 可以读状态并调用渲染接口，但不能修改玩法规则。
- `Services` 可以协调多个系统，但不直接绘制。
- `Data` 负责持久化格式，不负责解释玩法规则。
- 第三方类型要尽量限制在边界内，例如 Box2D 的 `b2BodyId` 后续应逐步被物理句柄或适配层包住。

## 推荐场景结构

```cpp
class IScene {
public:
    virtual ~IScene() = default;
    virtual void enter() = 0;
    virtual void handleEvent(const InputEvent& event) = 0;
    virtual void update(float dt) = 0;
    virtual void render(RenderContext& ctx) = 0;
    virtual void exit() = 0;
};
```

短期目标场景：

- `MainMenuScene`：主菜单输入、菜单视图、开始/读取/退出命令。
- `WorldScene`：当前游戏世界更新和渲染。
- `LoadingScene`：后续资源加载或区域切换过渡。
- `PauseScene`：后续暂停、设置、读档。

## 渐进式重构路线

### 阶段 1：降低 `main.cpp` 密度

- 抽出 `MainMenuView`，移走菜单绘制和点击命中计算。
- 抽出 `HudView`，移走 HUD 绘制。
- 抽出 `WorldRenderer`，移走 TileMap、装饰、角色、敌人、投射物、粒子渲染。
- 抽出 `InputState`，替代主循环内直接操作 SDL 键盘数组。

### 阶段 2：建立场景边界

- 引入 `IScene`、`SceneManager`。
- 将 `AppMode::MainMenu / Playing` 迁移为 `MainMenuScene / WorldScene`。
- 主循环只负责事件分发、更新时间、调用当前场景。

### 阶段 3：拆分运行时上下文

- 将 `GameState` 拆成 `WorldState`、`PlayerState`、`CombatState`、`UIState`、`RuntimeServices`。
- 系统只接收自己需要的上下文。
- 避免新增功能继续向 `GameState` 追加字段。

### 阶段 4：服务层重整

- 将 `saveCurrentGame`、`loadGameSlot`、`applySaveData` 迁移到 `SaveService`。
- 将区域切换和天气/小地图刷新迁移到 `RegionService`。
- 将对话启动、选择、奖励派发迁移到 `DialogueService`。

### 阶段 5：数据访问层重整

- 拆分 `SaveSystem` 为 `SaveService`、`SaveRepository`、`SaveSerializer`、`SaveSnapshotBuilder`、`SaveApplier`。
- 将 Lua 配置读取收口到 `LuaConfigRepository`。
- 为存档版本迁移保留 `SaveMigration`。

### 阶段 6：补齐音频层

- 新增 `Engine/Audio`。
- 接入 SDL_mixer、SoLoud 或 MiniAudio 之一。
- 先实现区域 BGM、技能 SFX、UI SFX。
- 后续通过事件驱动播放：`RegionEntered`、`SkillCast`、`EnemyKilled`、`WeatherChanged`。

### 阶段 7：CMake 模块化

- 将当前单一 `Starchild2D` target 拆成：

  ```text
  StarchildEngine
  StarchildGame
  StarchildApp
  ```

- `StarchildApp` 只链接并装配前两者。
- 为纯逻辑系统预留测试 target。

## 首轮重构验收标准

- `main.cpp` 行数下降，至少移除一个完整 UI 或渲染职责。
- 菜单、游戏启动、读档、退出行为保持不变。
- CMake Release 构建通过。
- 不修改 `build/` 生成物。
- 每一步重构都保持项目处于可编译状态。

## 代码现状快照（2026-06-09）

本节记录按阶段 1、阶段 2、阶段 4/5 对当前代码的对照结果，供后续继续重构时判断已完成边界和剩余风险。

### 阶段 1 现状：基本完成

已完成：

- `src/main.cpp` 已缩到创建 `Application` 并调用 `run()`，入口层目标基本达成。
- 菜单绘制与点击命中已抽到 `Game/Presentation/MainMenuView`。
- HUD 已抽到 `Game/Presentation/HudView`，并通过 ViewModel 渲染。
- TileMap、装饰等世界渲染已抽到 `Game/Presentation/WorldRenderer`，角色、敌人、投射物、粒子也拆到多个 Presentation 模块。
- 主渲染编排集中到 `Game/Presentation/GameRenderer`，不再堆在入口文件中。

未完成或未完全完成：

- 文档要求的 `InputState` 尚未真正抽出。当前仍是 `GameState::keys[SDL_NUM_SCANCODES]`，并且移动、玩具小游戏等系统直接读取该数组。
- 渲染职责虽然离开了 `main.cpp`，但 `GameRenderer` 仍直接读取完整 `GameState`，视图状态与玩法状态还没有完全隔离。

### 阶段 2 现状：已有场景雏形，但不是完整多态场景接口

已完成：

- 已存在 `Game/Scenes/SceneManager`、`MainMenuScene`、`WorldScene`。
- 主循环已变成事件轮询后调用当前场景更新与渲染。
- `MainMenu / Playing` 的更新渲染路径已分别进入 `MainMenuScene` 和 `WorldScene`。

未完成或未完全完成：

- 之前尚未引入文档示例中的 `IScene` 多态接口，也没有稳定的 `enter / handleEvent / update / render / exit` 生命周期。
- `SceneManager` 仍依赖 `GameState::appMode` 判断当前模式，场景切换还不是完全由场景管理器持有的多态对象驱动。
- 输入分发仍主要由全局 `InputController` 完成，场景只承担更新/渲染边界。

### 阶段 4/5 现状：保存与数据访问已部分提前推进

已完成或部分完成：

- 保存流程已拆出 `SaveGameService`、`SaveSnapshotBuilder`、`SaveApplier`。
- 文件读写和 JSON 序列化已拆到 `Game/Data/SaveRepository`、`SaveSerializer`。
- Lua 配置读取已有多个 Data Loader，例如天气、时间、情绪、家具、玩具、任务、对话脚本 Loader。

未完成或未完全完成：

- `SaveSystem` 仍存在并挂在 `GameState` 中，`SaveGameService` 仍通过 `gs.saveSystem` 间接读写，不是完全意义上的 `SaveService + Repository + Serializer` 组合。
- 读档套用存在状态回滚风险：重置会话时若不重建 `RegionManager`，旧会话访问过的区域可能污染旧存档读档结果。
- 旧任务存档 fallback 存在残留风险：只写入 `completedQuests` 时，如果不重建 `questEntries`，当前会话中已奖励的任务可能被错误保留。
- 没有独立 `RegionService`。区域切换、天气上下文、小地图刷新目前分散在 `RegionUpdateService` 和 `SessionService`。
- 没有独立 `DialogueService`。对话启动、选择、情绪奖励、公主好感仍主要在 `InteractionInputController` 中直接处理。
- 没有统一的 `LuaConfigRepository`。目前是多个 Loader 加上系统自己的 `loadConfig/loadDefinitions`，路径仍在 `GameBootstrap` 和部分服务中直接传入。
- 没有 `SaveMigration`，存档版本字段只是读写保存，尚未形成迁移层。

### 当前下一步优先级

1. 先修复读档状态污染：读档前重建 `RegionManager`，避免已访问区域、缓存区域和发现区域污染旧存档。
2. 修复旧任务存档 fallback：加载 completed quest 列表前重建所有任务条目，避免缺失任务被错误保留为 Rewarded。
3. 补齐阶段 2 的最小 `IScene` 边界，使场景至少拥有明确的 `enter / handleEvent / update / render / exit` 生命周期。
4. 后续进入阶段 3，拆分 `GameState` 为更小上下文，再继续收口 `RegionService`、`DialogueService`、`LuaConfigRepository` 和 `SaveMigration`。
